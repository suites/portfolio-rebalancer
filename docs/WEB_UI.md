# Portfolio Rebalancer Web GUI 설계

## 1. 문서 목적

이 문서는 Portfolio Rebalancer를 Web GUI로 운영하기 위한 정보 구조, 화면, 공통 컴포넌트, 사용자 흐름과 접근성 요구사항을 정의합니다.

Web GUI의 목적은 빠른 매매가 아닙니다. 사용자가 다음 세 질문에 즉시 답을 얻는 **안전한 자산관리 운영 콘솔**을 만드는 것이 목표입니다.

1. 지금 시스템과 포트폴리오가 정상인가?
2. 왜 리밸런싱이 필요한가?
3. 실행하면 무엇이 어떻게 바뀌는가?

## 2. 설계 원칙

- 트레이딩 터미널이 아니라 장기 자산관리 콘솔로 설계합니다.
- 첫 화면에서는 수익률보다 현재 상태와 필요한 행동을 강조합니다.
- 내부 주문 상태, API 응답과 DB 구조는 고급 상세정보로 숨깁니다.
- 계획 생성, 설정 변경과 주문 실행을 서로 다른 행동으로 분리합니다.
- 실거래 전에는 이유, 주문, 비용, 위험 검사와 변경 후 비중을 다시 확인합니다.
- 장애 시 강제 진행보다 거래 차단과 안전한 복구를 우선합니다.
- 색상은 보조 수단으로만 사용하고 상태명, 아이콘과 설명을 함께 제공합니다.
- 사용자가 YAML이나 SQLite를 직접 수정해야 정상 운영할 수 있는 GUI는 완성으로 간주하지 않습니다.

### 2.1 시스템 경계

Web GUI는 별도의 리밸런서가 아니라 기존 애플리케이션의 안전한 운영 인터페이스입니다. CLI와 GUI는 동일한 애플리케이션 서비스, 계산기, Risk Gate와 주문 원장을 사용합니다. 브라우저는 토스증권 API를 직접 호출하지 않고, 비중 계산이나 주문 판단을 수행하지 않으며, 서버가 확정한 계획과 상태만 표시합니다.

현재 생산 Web GUI도 이 경계를 따릅니다. `apps/engine`이 실제 토스 조회를 PostgreSQL에 저장하고, `apps/web/src/server`가 engine 응답을 Zod 계약으로 재검증한 뒤 클라이언트 컴포넌트에 전달합니다. 브라우저는 토스와 PostgreSQL에 직접 접근하지 않습니다.

### 2.2 디자인 기준 산출물

Web GUI의 기준은 다음 계층으로 관리합니다.

| 기준             | 파일                     | 역할                                             |
| ---------------- | ------------------------ | ------------------------------------------------ |
| 행동 기준        | `docs/WEB_UI.md`         | 정보 구조, 사용자 흐름, 상태, 안전과 접근성 계약 |
| 생산 시각 기준   | `packages/ui/src/styles` | primitive·semantic·component 토큰과 전역 스타일  |
| 생산 컴포넌트    | `packages/ui/src`        | 공통 UI의 props, 상태와 마크업 계약              |
| 토큰 호환 진입점 | `design/tokens.css`      | 생산 토큰 계층을 기존 경로에 연결                |
| 탐색 참고자료    | `prototype/index.html`   | 합성 시나리오의 초기 레이아웃·상태 탐색물        |

프로토타입은 실제 금융 계산이나 주문 판단을 수행하지 않는 참고자료입니다. 생산 구현과 충돌하면 안전·행동은 이 문서, 시각 값과 컴포넌트 계약은 `packages/ui`를 우선합니다. Storybook은 아직 구성되지 않았으며 도입 후 컴포넌트 상태의 실행 가능한 문서 기준으로 추가합니다.

### 2.3 현재 구현 범위

2026-07-16 기준 생산 화면은 토스 실제 데이터 기반 Shadow·Paper·제한형 Live 운영
콘솔입니다. Live 코드 구현과 실계좌 운영 승격 완료는 별개이며, 실제 극소액 주문
검증은 아직 수행하지 않았습니다.

- Next.js App Router와 server-only 조합 계층
- 운영 설정 원장에서 읽은 PAPER/LIVE 모드, 마스킹 계좌, 수집 시각, 킬 스위치와 Live 승격 상태
- `토스증권 실제 데이터`와 실제 engine 연결 상태를 명시
- 6개 실제 링크, 현재 경로 표시와 모바일 2~3열 내비게이션
- 홈의 상태 결론 배너, 자산 요약, 금액 숨김과 최근 상태
- 홈에서 주문 없이 최신 토스 보유자산·가격을 다시 수집하는 명시적 동작
- 포트폴리오의 비중 밴드와 caption·column header가 있는 native 보유자산 표
- 리밸런싱의 현재/목표 비교, 저장된 주문 후보와 예상 비중
- SHADOW/PAPER/LIVE 계획 생성과 설정·실행을 분리한 명시적 동작
- Paper 주문 원장 실행과 실제 호가 증거 기반 모의 체결
- Live 확인 문구, 주문별 승인 생성과 첫 주문 한 건의 one-shot dispatch
- append-only 주문 타임라인, 브로커 상태 재확인, 명시적 취소와 `UNKNOWN_BLOCKED` exact 복구
- 승인 뒤 dispatch가 전혀 없음을 DB가 증명한 경우의 미전송 `REJECTED` 복구
- 실제 계좌 수집·스냅샷 검사 기록
- 문제·보호 조치·다음 행동 진단과 사용자가 요청하는 read-only 재점검
- 현재 보유종목을 `SAFE/CORE/SATELLITE`로 분류하는 snapshot-bound 자산군 목표 DRAFT 저장, 별도 적용과 새 스냅샷 요구 상태
- 자산군 내부 현재 평가액 비율을 `PRESERVE_CURRENT_V1`로 고정하고 검토 화면에 구성 종목 표시
- 이전 Toss 검증 결과로 만든 로컬 카탈로그의 종목명 검색과 국내 코드·미국 티커의 Toss 정확 검증을 분리
- 목표 편입 가능 여부와 현재 거래 차단 여부를 구분해 표시하고 검증된 미보유 종목을 자산군에 추가
- 미보유 종목이 포함된 자산군의 `EQUAL_V1` 명시적 균등 배분
- 목표 비중만 입력하고 `MIXED_V1` 정책으로 하한·상한을 서버에서 자동 계산
- 같은 0~100% 축에서 현재·목표·허용 범위를 표시하는 `AllocationBand`
- 비중 이탈은 `확인 필요`로, 데이터·주문 차단은 `차단`으로 구분
- 표시용 0.01bp 단위와 별개로 서버에서 `bigint` 교차곱으로 1bp 미만 이탈 판정
- 목표 비중 미설정 자산을 가짜 밴드 없이 표시하고 설정 화면으로 연결
- 설정에 없는 현재 보유종목을 별도 목록으로 표시하고 자동 매도 없이 거래 차단
- 설정 불일치, 관리 현금 정책 미설정·미반영, 미해결 주문과 안전 증거 결손에서 실행 차단
- 총 관리 자산, 사용자 지정 관리 현금과 토스 매수 가능 금액을 서로 다른 값으로 표시
- 설정에서 관리 현금을 고정 원화 금액으로 포함하거나 평가에서 제외하고 `CASH` 목표를 별도 입력
- 운영 설정 DRAFT 저장·해시 검토·별도 적용, 현재 계좌 서버 봉인, 킬 스위치와 Live 승격
- Tailscale 내부망·Caddy TLS와 engine loopback 접근 경계
- primitive·semantic·component 토큰과 `Button`, `Badge`, `Surface`, `StatusBanner`, `SummaryCard`, `AllocationBand`
- 공통 Button·StatusBanner·AllocationBand의 정적 렌더링 계약 테스트

상단 safety bar는 포트폴리오 결론과 별개로 실제 운영 설정의 모드, 킬 스위치와
`실주문 조건 충족/차단`을 표시합니다. 이 표시는 주문별 승인과 pretrade 검사를 대신하지
않습니다. 모든 내비게이션 항목은 실제 화면으로 이동합니다. 설정 초안은 저장 당시
스냅샷에 묶이며 데이터가 바뀌면 적용하지 않습니다. 목표 설정 저장·적용, 계획 생성과
주문 실행은 서로 다른 동작입니다.

주문·기록은 실제 `OrdersSnapshot`만 사용하며 수집 기록과 주문 원장을 분리해 보여
줍니다. 취소는 브로커 ID가 확인된 Live `PENDING/PARTIAL_FILLED`에서만 일회성
authorization과 dispatch claim을 거쳐 전송합니다. `UNKNOWN`, `UNKNOWN_BLOCKED`,
부분체결과 미체결 주문에 수동 재제출 동작을 제공하지 않습니다. 전용 감사 증거
viewer, 실행 영수증 상세, doctor/explain 통합과 자동 Phase B 재계획은 후속 범위입니다.

| 화면              | 사용 목적                                  |

| 홈                | 오늘의 상태와 필요한 행동 확인             |
| 포트폴리오 만들기 | 투자성향별 추천 종목과 비중 확인·초안 저장 |

흐름에서 필요한 경우 홈과 리밸런싱 화면이 해당 상세 경로로 문맥 링크를 제공합니다. 종목 직접
검색, 자산군 분류, 관리 현금, 운영 한도, 킬 스위치와 Live 승격은 `포트폴리오 만들기`의
고급 설정 안에 둡니다.

운영자용 전체 정보 구조는 다음과 같습니다.

| 포트폴리오 | 현재·목표 비중과 보유자산 확인               |

평상시 사용은 `홈 → 리밸런싱 계획 → 실행 결과` 안에서 끝나야 합니다.

### 3.1 포트폴리오 만들기 기본 흐름

1. 현재 총 관리 자산을 읽기 전용으로 확인합니다.
2. 안정형·균형형·성장형 중 하나를 선택합니다. 기본 선택은 균형형입니다.
3. 최대 7개 이내의 승인된 종목, 목표 비중과 각 종목의 역할을 확인합니다.
4. 추천 밖 기존 보유종목이 있으면 목표 0% 정리 후보임을 별도 경고로 확인합니다.
5. `이 추천안으로 초안 만들기`를 눌러 저장합니다. 이 단계에서는 적용이나 주문이 없습니다.
6. 버전 검토에서 초안을 별도로 적용한 뒤 최신 계좌를 다시 수집하고 리밸런싱 계획을 확인합니다.

기본 화면은 개별 종목 검색, 자산군 선택, CASH 비중, 내부 배분 정책과 밴드 입력을 노출하지
않습니다. 추천 엔진은 코드에 승인된 버전형 종목만 사용하며 저장 시 서버가 종목을 다시
검증합니다. 향후 AI는 이 승인 유니버스 안의 후보 정렬과 쉬운 설명만 담당하고 적용·주문 권한을
갖지 않습니다.

## 4. 전역 App Shell

모든 화면 상단에 `GlobalSafetyBar`를 고정합니다.

```text
[모드: PAPER] [계좌: ****1234] [데이터: 2분 전]
[시장: 장중] [시스템: 정상] [킬 스위치: 해제]
```

필수 정보:

- 실행 모드: Shadow / Paper / Live
- 마스킹된 계좌
- 시장 상태: 장중 / 장외 / 휴장 / 확인 불가
- 데이터 기준 시각과 경과 시간
- 시스템 상태: 정상 / 확인 필요 / 차단 / 확인 중 / 알 수 없음
- 킬 스위치 상태

Live 모드는 색상뿐 아니라 `실거래`, 아이콘과 테두리로 명확히 구분합니다. 시스템이나 데이터 상태가 불확실하면 실행 버튼을 자동으로 비활성화합니다.

## 5. 홈

홈은 자산 규모보다 `지금 무엇을 해야 하는가`를 먼저 보여줍니다.

```text
┌──────────────────────────────────────────────────────────┐
│ ✓ 정상 — 오늘은 거래할 필요가 없습니다                   │
│ 마지막 점검 09:12 KST · 다음 점검 내일 09:00             │
└──────────────────────────────────────────────────────────┘

┌──────────────┐ ┌──────────────┐ ┌─────────────────┐
│ 총 관리 자산 │ │ 운용 가능 현금│ │ 범위 밖 자산 0개│
│ ₩10,000,000  │ │ ₩1,000,000   │ │ 정상            │
└──────────────┘ └──────────────┘ └─────────────────┘

현재 비중과 목표
광범위 시장     74% / 목표 75% / 정상
AI 반도체       16% / 목표 15% / 정상
현금            10% / 목표 10% / 정상

최근 활동
09:12  계좌 점검 완료 — 주문 없음
어제   Discord 알림 전송 완료
```

### 5.1 홈 상태 결론

가장 큰 상태 카드는 다음 중 하나만 표시합니다.

- 정상 — 거래 필요 없음
- 계획 있음 — 검토 필요
- 실행 완료
- 거래 차단 — 확인 필요
- 점검 실패 — 진단 필요

내부 상태 코드인 `NO_ACTION`, `BLOCKED`, `UNKNOWN_BLOCKED`는 고급 상세정보에서만 표시합니다.

### 5.2 홈 컴포넌트

- `SystemStatusBanner`
- `PortfolioSummaryCards`
- `AllocationSummary`
- `ActionRequiredCard`
- `RecentActivityList`
- `NextRunIndicator`
- `AmountPrivacyToggle`

홈 헤더의 `최신 자산 가져오기`는 토스 보유자산·가격·환율·매수 가능 금액을 다시
조회해 새 불변 스냅샷으로 저장한 뒤 홈의 수집 시각과 자산 상태를 갱신합니다. 이 동작은
계획 생성이나 주문 제출을 수행하지 않으며, 실패하면 기존 스냅샷을 덮어쓰지 않고 거래를
차단한 상태에서 문제 해결 화면으로 안내합니다.

## 6. 포트폴리오

포트폴리오 화면은 도넛 차트보다 목표 범위가 함께 표시되는 가로 밴드와 표를 우선합니다.

| 자산군      | 현재 | 목표 |    허용 범위 |     차이 | 상태          |
| ----------- | ---: | ---: | -----------: | -------: | ------------- |
| 광범위 시장 |  68% |  75% |       70~80% | 부족 7%p | 리밸런싱 필요 |
| AI 반도체   |  22% |  15% | 11.25~18.75% | 초과 7%p | 리밸런싱 필요 |
| 현금        |  10% |  10% |    7.5~12.5% |      0%p | 정상          |

- `AllocationBandBar`: 현재 비중, 목표선과 허용 범위
- `AllocationTable`: 정확한 수치와 상태

차트는 보조 수단입니다. 동일한 정보는 항상 텍스트와 접근 가능한 표로 제공합니다.

생산 `AllocationBand`의 범위 이탈은 거래 불가를 뜻하는 `blocked`가 아니라 검토가 필요한 `attention` 상태입니다. 화면에는 현재 비중을 0.01bp 단위까지 전달하지만, 서버의 실제 판정은 자산 평가액과 전체 평가액을 `bigint`로 교차 비교해 그보다 작은 이탈도 놓치지 않습니다. 서버가 전달한 `bandStatus`가 표시 상태의 기준입니다.

## 7. 리밸런싱 계획

가장 중요한 화면이며 주문 목록보다 `왜 거래하는가`를 먼저 설명합니다.

> AI 반도체 비중이 허용 상한 18.75%를 3.25%p 초과했습니다. 일반 정책에 따라 목표 15%가 아니라 허용 범위 경계인 18.75%까지 조정합니다.

### 7.1 Before / After 비교

```text
                현재       실행 후 예상
광범위 시장      68%   →       72%
AI 반도체        22%   →       18%
현금             10%   →       10%
```

### 7.2 주문 계획 표

| 구분 | 종목       | 수량 | 기준가격 | 예상금액 | 예상비용 | 이유           |
| ---- | ---------- | ---: | -------: | -------: | -------: | -------------- |
| 매도 | 반도체 ETF |  5주 |  ₩10,000 |  ₩50,000 |      ₩15 | 비중 상한 초과 |
| 매수 | 시장 ETF   |  2주 |  ₩24,000 |  ₩48,000 |      ₩14 | 목표 비중 부족 |

표시 항목:

- 총 예상 매수·매도 금액
- 예상 수수료와 슬리피지
- 예상 잔여 현금
- 일일 거래 한도 사용률
- 계획 유효기간과 데이터 기준 시각
- 실제 체결 결과가 달라질 수 있다는 설명

### 7.3 위험 검사 체크리스트

```text
✓ 계좌 조회 성공
✓ 가격 데이터가 최신입니다
✓ 기존 미체결 주문이 없습니다
✓ 일일 거래 한도 이내입니다
✓ 주문 후 종목 최대 비중을 넘지 않습니다
✓ 매도 가능 수량과 매수 가능 금액을 확인했습니다
```

검사 상태는 `통과`, `확인 중`, `경고`, `실행 불가`로 제한합니다. 실행 불가 규칙에 bypass 버튼을 제공하지 않습니다.

### 7.4 계획 동작

- 계획 다시 계산
- 계획 폐기
- Paper 주문 실행
- 실제 주문 검토

사용자가 개별 주문 행을 임의로 수정하면 전체 위험 계산이 무효화되므로 자유 형식 주문 편집기는 제공하지 않습니다.

## 8. 실거래 확인과 영수증

계획 생성과 주문 제출은 서로 다른 화면과 행동으로 분리합니다.

최종 확인 화면은 다음을 다시 보여줍니다.

- `실거래` 모드
- 대상 계좌
- 이번에 전송할 첫 주문과 전체 계획의 남은 주문 수
- 총 예상 거래금액과 비용
- 데이터 기준 시각
- 실제 체결가격과 결과가 달라질 수 있다는 안내

버튼은 `확인`이 아니라 결과를 구체적으로 작성합니다.

```text
[취소] [승인 생성 후 Live 첫 주문 1회 전송]
```

현재 생산 UI는 계획 표 아래에서 대상 계좌, 이번에 전송할 첫 주문, 남은 주문 수,
전체 계획 예상 거래금액, 비용 재검증 방식과 데이터 기준 시각을 다시 보여 주고 exact
확인 문구를 요구합니다. 매도 우선 첫 주문 한 건만 전송하며 다음 주문은 체결 확인,
새 snapshot과 새 계획 뒤 다시 승인해야 합니다. 제출 후에는 주문·기록 링크와 현재
상태를 제공하며, 다음 항목을 한 화면에 묶은 전용 `ExecutionReceipt`는 후속
구현입니다.

- 실행 ID와 주문 접수번호
- 접수 시각과 현재 상태
- 다음 확인 예정 시각
- 취소 가능 여부
- 주문·체결 상세 링크

## 10. 문제 해결과 안전장치

이 화면의 목표는 CLI의 `doctor`, `explain`, `recover`를 GUI로 제공하는 것입니다.

주문 상태 재확인·취소·`UNKNOWN_BLOCKED` 복구는 주문·기록 화면에 구현되어 있습니다.

- 마지막 예약 실행

오류 카드는 다음 구조로 통일합니다.

기본 화면에는 관리 현금 정책, 현재 보유종목과 사용자가 추가한 미보유 종목의 자산군 분류, 네 자산군 목표 비중, 자동 계산된 허용 범위와 자산군 내부 배분 방식만 표시합니다. 사용자는 기본 흐름에서 하한·상한이나 개별 종목 내부 비중을 직접 입력하지 않습니다. 관리 현금은 `고정 관리금액 포함` 또는 `포트폴리오 평가에서 제외` 중 하나를 명시적으로 선택하며, 전자는 원 단위 비음수 정수를 요구하고 후자는 `CASH` 목표 0%만 허용합니다. 모든 현재 보유종목은 `안전자산`, `핵심 공격자산`, `위성 공격자산` 중 정확히 한 곳에 배치해야 하며 제거할 수 없습니다. 사용자가 추가한 현재 미보유 종목만 목표 구성에서 제거할 수 있습니다.

종목 찾기와 목표 초안 저장은 별도 동작입니다. 종목명 검색은 `GET /internal/v1/instruments/search`를 통해 서버가 이전에 Toss로 검증해 둔 `LOCAL_VALIDATED` 카탈로그만 조회합니다. 국내 6자리 코드, 미국 티커와 `KR:/US:` 형식은 `POST /internal/v1/instrument-validations`로 Toss 기본정보와 유의사항을 정확히 재검증합니다. 검색이나 검증만으로 설정 버전을 만들거나 적용하지 않습니다. 검색 결과에는 다음을 구분해 표시합니다.

- 상장 상태
- 장기 목표 편입 가능 여부와 차단 이유
- 현재 거래 차단 여부와 재검증 필요 안내
- 검증 기준 시각

목표 편입이 차단된 결과에는 추가 동작을 제공하지 않습니다. 현재 거래만 차단된 종목은 장기 목표 후보로 추가할 수 있더라도, 향후 계획 생성과 주문 직전에 유의사항을 다시 확인해야 한다는 설명을 함께 표시합니다.

현재 보유종목만 있는 자산군은 `현재 평가액 비율 유지(PRESERVE_CURRENT_V1)` 또는 `균등 배분(EQUAL_V1)`을 선택할 수 있습니다. 미보유 종목이 하나라도 포함된 자산군은 평가액 기반 비중을 계산할 수 없으므로 `EQUAL_V1`을 명시적으로 선택해야 합니다. 균등 배분의 나머지 점수는 정규 종목 키 순서로 결정해 합계를 정확히 10000점으로 고정합니다. 현재 보유종목의 평가액 합계가 0인 자산군은 `PRESERVE_CURRENT_V1`로 저장하지 않습니다.

검색 오류, Toss 검증 실패, 목표 합계 오류 또는 초안 저장 실패가 발생해도 입력한 관리 현금, 목표 비중, 자산군 선택과 추가한 미보유 종목을 보존합니다. 오류는 폼 안의 지속되는 텍스트로 원인과 다음 행동을 알리고, 저장 성공 후에만 서버의 새 버전을 다시 읽습니다. 수동 허용 범위, 심각한 이탈 기준, 슬리피지, 데이터 freshness와 반올림 정책은 고급 설정으로 분리합니다. 토스의 `매수 가능 금액`은 주문 가능성 참고값으로 표시하되 `관리 현금` 또는 `총 관리 자산`으로 이름을 바꾸거나 자동 합산하지 않습니다.

목표 비중 합이 100%가 아니면 문제만 알리지 말고 수정 방법을 제안합니다.

```text
목표 비중 합계가 95%입니다.
5%를 추가로 배분하거나 현금 비중을 10%에서 15%로 조정하세요.
```

설정 저장과 주문 실행을 연결하지 않습니다. 설정이 바뀌면 기존 계획을 무효화하고 새 거래 영향을 검토해야 합니다.

| ------------------------------- | ------------------------------ |
| 정상 — 거래 필요 없음           | `NO_ACTION`                    |
| 리밸런싱 필요 — 계획 미생성     | 화면 계약 `REBALANCE_REQUIRED` |
| 주문 계획 있음 — 검토 필요      | `PLANNED`                      |
| 실행 완료                       | `EXECUTED`, `FILLED`           |
| 일부 체결 — 확인 중             | `PARTIAL_FILLED`               |
| 거래 차단                       | `BLOCKED`, `UNKNOWN_BLOCKED`   |
| 결과 확인 필요 — 신규 주문 차단 | `UNKNOWN`                      |
| 점검 실패                       | `FAILED`                       |

`UNKNOWN`은 단순 실패로 표현하지 않습니다. 주문 결과를 확인할 때까지 신규 주문과 수동 재제출이 차단된다는 설명을 항상 함께 표시합니다.

권장 의미 토큰:

```text
status.normal
status.attention
status.blocked
status.pending
status.unknown
```

색상에 더해 상태 아이콘, 제목과 설명을 항상 제공합니다. 상승·하락과 매수·매도도 빨강·초록만으로 구분하지 않습니다.

## 14. 숫자와 시간 표기

- 숫자는 우측 정렬하고 tabular numerals를 사용합니다.
- 금액은 `₩1,234,567`, `$1,234.56`처럼 통화를 표시합니다.
- 비중은 `67.0%`, 편차는 `초과 7.0%p`처럼 단위를 구분합니다.
- `1.2M` 같은 축약은 보조 표시에만 사용하고 전체 값을 함께 제공합니다.
- 합산 평가에는 기준 통화, 환율과 환율 기준 시각을 표시합니다.
- 시간은 `오늘 15:30 KST · 47분 전`처럼 절대·상대 시각을 함께 표시합니다.

## 15. 접근성 요구사항

제품 기준은 WCAG 2.2 AA입니다.

- 일반 텍스트 대비 4.5:1 이상
- 큰 텍스트와 의미 있는 비텍스트 요소 대비 3:1 이상
- 모든 기능을 키보드만으로 수행 가능
- 항상 보이는 focus indicator
- 고정 헤더와 토스트가 현재 포커스를 가리지 않음
- 44×44 CSS px를 권장 조작 크기로 사용
- 네이티브 `button`, `input`, `table`, landmark와 heading 우선
- 차트 앞에 텍스트 요약을 두고 동일 데이터를 표로 제공
- 상태 갱신은 `aria-live`로 알리되 가격 변화마다 읽지 않음
- 중요한 오류는 자동 소멸 토스트만으로 전달하지 않음
- `lang="ko"`와 명확한 accessible name 제공

폼 오류는 상단 `ErrorSummary`와 필드 인라인 메시지에 동시에 표시합니다. 오류 메시지는 `무엇이 잘못됐는지`와 `어떻게 수정하는지`를 설명하며 입력값을 보존합니다.

접근성 검증:

- axe 등 자동 검사
- 전체 키보드 흐름 수동 검사
- macOS VoiceOver와 Safari 검사
- 200% 확대와 320 CSS px reflow
- forced-colors와 색각 시뮬레이션
- 설정, 계좌 점검, 계획 검토, Paper 실행, 실거래 확인과 UNKNOWN 복구 E2E
- 새 사용자가 GUI만으로 설정, 주문 없는 점검, 계획 검토, Paper 실행과 결과 확인을 완료하는 사용성 검사

현재 생산 기반에는 skip link의 포커스 복구, 링크·버튼의 `focus-visible`, 44px 최소 조작 높이, reduced-motion과 forced-colors 대응을 적용했습니다. Button의 native disabled, StatusBanner의 고유 accessible ID와 AllocationBand의 텍스트 범위 설명을 정적 렌더링 테스트로 확인합니다. 이는 전체 WCAG 2.2 AA, 키보드와 VoiceOver 검증 완료를 의미하지 않습니다.

## 16. Web 보안 요구사항

- 기본 배포는 단일 호스트의 localhost 또는 명시적으로 보호된 사설 네트워크로 제한합니다.
- 토스 client secret, access token과 Discord webhook은 서버에서만 보관하며 HTML, API 응답, 브라우저 저장소와 콘솔 로그에 포함하지 않습니다.
- Host-run 접근 경계는 Tailscale 내부망, Caddy TLS와 `127.0.0.1`에만 bind한 engine으로 구성합니다. 일반 Web→Engine 호출에는 별도 애플리케이션 인증을 추가하지 않습니다.
- 계획 생성과 주문 제출 API를 분리하고, 페이지 새로고침·뒤로 가기·중복 클릭·네트워크 재시도가 주문 재제출로 이어지지 않게 합니다.
- 실행 버튼은 제출 직후 비활성화하되 UI 상태만 믿지 않고 서버의 논리 주문 고유성과 상태 검사를 사용합니다.
- 새 스냅샷, 설정 변경 또는 계획 만료가 발생하면 기존 계획을 실행 불가로 표시합니다.
- Live 주문, 킬 스위치 해제와 복구처럼 위험한 행동에는 정확한 확인 문구, 만료되는 주문별 승인과 서버 Risk Gate를 적용합니다.
- 사용자 승인, 설정 변경, 복구와 킬 스위치 변경을 감사 로그에 남깁니다.
- CSP와 최소 권한 원칙을 적용하고 외부 스크립트·분석 도구가 금융 데이터나 식별자를 수집하지 않게 합니다.

현재 Web의 `dev`와 `start` 명령은 `127.0.0.1:13000`, engine은 `127.0.0.1:4100`에 고정합니다. host-run production은 Caddy TLS와 Tailscale DNS-only A record를 통해 `stock.fredly.dev`로 Web만 제공하되 공용 인터넷에는 노출하지 않습니다. Vercel에서는 Deployment Protection 없이 engine을 공개하는 배포를 지원하지 않습니다.

## 17. 반응형 정책

- 데스크톱을 전체 운영 기능의 기준으로 합니다.
- 태블릿은 점검, 계획 검토와 승인을 지원합니다.
- 모바일은 우선 읽기 전용 상태 확인과 알림 진입에 집중합니다.
- 화면의 최상위 카드와 섹션은 공통 간격 토큰을 사용하며, 모바일에서도 서로 맞닿지 않게 동일한 수직 리듬을 유지합니다.
- 작은 화면에서 표는 핵심 열을 유지하고 상세정보를 행 확장 또는 카드로 제공합니다.
- 가로 스크롤에만 의존하지 않고 320 CSS px에서 핵심 흐름이 reflow 되어야 합니다.

## 19. 구현 우선순위

Web GUI는 다음 수직 흐름 순서로 구현합니다.

1. `AppShell`, 상태 토큰과 Storybook 기반 컴포넌트 토대
2. 읽기 전용 홈과 포트폴리오
3. 리밸런싱 이유, Before/After와 위험 검사
4. 설정 마법사와 설정 변경 영향 미리보기
5. Paper 실행, 주문 타임라인과 영수증
6. 문제 해결, 진단과 복구
7. 접근성 E2E와 실거래 확인 흐름

현재는 App Shell, 실제 포트폴리오, 계획 표와 예상 비중, 목표·운영 설정, Paper 실행,
제한형 Live 확인, 주문 상태 머신, 취소·대사·복구, 킬 스위치까지 생산 구현되어 있습니다. 설정 변경 Diff와
거래 영향 미리보기, 전체 Risk Gate 증거 표시, 전용 실행 영수증·감사 상세,
doctor/explain, 자동 Phase B saga, CSP, WCAG와 전체 브라우저 E2E는 후속 범위입니다.

첫 번째 핵심 수직 슬라이스는 `계좌 스냅샷 → 리밸런싱 계획 → 위험 검사 → 주문 없는 결과 확인`입니다. 첫 구현 화면은 홈보다 리밸런싱 계획 검토 화면을 우선해도 좋습니다. 이 화면이 계산, 설명, 위험 차단과 실행 계약을 모두 연결하기 때문입니다.

## 20. 완료 기준

- 사용자가 10초 안에 현재 상태와 필요한 행동을 이해할 수 있다.
- 정상적인 날은 홈 확인만으로 사용이 끝난다.
- 거래 전에 이유, 주문, 비용, 위험 검사와 거래 후 비중을 확인할 수 있다.
- 설정 변경이 만들 거래 영향을 적용 전에 볼 수 있다.
- 실거래와 Paper가 색상 외 텍스트·아이콘으로 구분된다.
- 불명확한 주문 상태에서 재주문이 차단된다.
- DB나 YAML 직접 수정 없이 상태 확인과 복구가 가능하다.
- 주요 사용자 흐름이 키보드와 스크린리더로 완료된다.
- 중요한 결과는 토스트 외에 주문 영수증과 감사 기록에 남는다.
- 새로고침, 뒤로 가기, 중복 클릭과 네트워크 재시도가 주문을 중복 제출하지 않는다.
- 비밀정보가 브라우저 저장소, HTML, 네트워크 응답과 클라이언트 로그에 노출되지 않는다.
