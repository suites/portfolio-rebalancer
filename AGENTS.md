# AGENTS.md

## 프로젝트 목적

이 저장소는 토스증권 Open API를 활용해 장기 목표 비중을 관리하는 개인용 자산배분 시스템입니다. 수익 예측이나 종목 발굴보다 결정론적 계산, 장애 안전성, 주문 멱등성과 감사 가능성을 우선합니다.

최종 품질 기준은 실제 사용입니다. 내부 안전장치를 줄여 사용성을 얻지 말고, 복잡성을 `setup`, `check`, `plan`, `run`, `status`, `explain`, `recover` 인터페이스 뒤에 감추세요. 사용자가 YAML 구조, 주문 상태 머신 또는 SQLite를 알아야 정상 운영할 수 있는 구현은 완료로 간주하지 않습니다.

작업 전 `README.md`, `docs/SPEC.md`, `docs/TODO.md`를 순서대로 읽으세요. Web GUI 또는 사용자 흐름을 변경할 때는 `docs/WEB_UI.md`도 읽으세요. 요구사항이 충돌하면 `docs/SPEC.md`를 기준으로 삼고, 금융 안전성에 영향을 주는 변경은 문서와 테스트를 함께 갱신하세요.

## 변경 원칙

- 종목 선정과 리밸런싱을 별도 도메인으로 유지하세요.
- 계산기와 위험 차단기는 브로커 API나 시스템 시각에 직접 의존하지 않는 순수 로직으로 작성하세요.
- 외부 데이터는 스냅샷으로 고정한 뒤 계산에 전달하세요.
- 금액과 수량 계산에 부동소수점 자료형을 사용하지 마세요.
- 모든 시간에는 명시적인 타임존을 사용하세요.
- 기본 실행 모드는 항상 `paper`여야 합니다.
- 실거래 활성화 조건을 약화하거나 하나의 플래그로 축소하지 마세요.
- API 오류, 누락 데이터 및 상태 불명 주문에서는 fail closed 하세요.
- 주문 계획은 저장한 뒤 실행하고, 상태 변경을 append-only로 기록하세요.
- 동일한 논리 주문은 로컬 `logical_order_id` UNIQUE 제약으로 보호하고 결정적인 `clientOrderId`를 보조 수단으로 사용하세요.
- 토스 `clientOrderId`의 서버 멱등성은 10분뿐이므로 10분 이후 불명확한 주문을 자동 재제출하지 마세요.
- `buying power`를 검증 없이 포트폴리오의 평가용 현금으로 사용하지 마세요.
- Web GUI와 CLI는 동일한 애플리케이션 서비스, 계산기, Risk Gate와 주문 원장을 사용하세요. 금융 계산이나 주문 판단을 프런트엔드에 복제하지 마세요.
- 브라우저에서 토스증권 API를 직접 호출하거나 토스 비밀정보를 전달하지 마세요.
- 설정 저장, 계획 생성과 주문 제출을 서로 다른 동작으로 유지하세요.
- `UNKNOWN`, 부분체결 또는 기존 미체결 주문에 수동 재제출 버튼을 제공하지 마세요.

## 금지 사항

명시적인 사용자 승인과 별도의 실거래 설계 검토 없이 다음을 수행하지 마세요.

- 실제 주문 엔드포인트 호출
- live 모드를 기본값으로 변경
- 주문 한도, 킬 스위치 또는 멱등성 검사를 우회
- 계좌 전체에 영향을 주는 자동 청산 기능 추가
- 뉴스, 가격 예측 또는 모델 출력으로 승인 종목 목록을 자동 변경
- 레버리지, 인버스, 옵션 또는 공매도 지원 추가

## 비밀정보

- API client secret, access token, 계좌번호, Discord webhook을 저장소에 커밋하지 마세요.
- 비밀정보는 환경변수 또는 승인된 secret store에서만 읽으세요.
- 로그, fixture, 스냅샷, 오류 메시지 및 알림에서 민감정보를 마스킹하세요.
- 실제 API 응답을 fixture로 추가할 때 모든 식별자를 합성값으로 교체하세요.

## 테스트 요구사항

다음 변경에는 테스트가 필수입니다.

- 비중, 환율, 수수료, 수량 반올림 계산
- 밴드 경계와 목표 복귀 정책
- 설정 검증과 허용 목록
- 위험 차단 규칙
- 주문 상태 전이와 재시작 복구
- `clientOrderId` 생성과 중복 실행
- 10분 멱등성 경계와 `UNKNOWN_BLOCKED`
- 일일 누적 한도 예약의 동시성
- 토스 API 오류 및 rate limit 처리
- 비밀정보 마스킹

핵심 로직은 네트워크 없이 테스트할 수 있어야 합니다. 외부 API 테스트는 기록된 합성 fixture 또는 명시적인 테스트 환경을 사용하세요.

## 문서 규칙

- 시스템 동작이나 안전 조건을 변경하면 `docs/SPEC.md`를 갱신하세요.
- 구현 단계가 끝나면 `docs/TODO.md`의 상태를 갱신하세요.
- 사용 방법이나 프로젝트 상태가 달라지면 `README.md`를 갱신하세요.
- 확인되지 않은 토스 API 동작을 사실로 작성하지 말고 `[확인 필요]`로 표시하세요.
- Web GUI의 정보 구조, 컴포넌트 상태 또는 접근성 계약이 바뀌면 `docs/WEB_UI.md`를 갱신하세요.

## Git 커밋 규칙

- 파일 변경이 발생한 작업은 검증을 마친 뒤 작업 종료 전에 반드시 커밋하세요.
- 각 작업의 변경은 해당 작업에 맞는 하나 이상의 Conventional Commit으로 정리하고, 커밋하지 않은 변경을 남긴 채 완료를 보고하지 마세요.
- 읽기·설명·검토처럼 파일 변경이 없는 작업에는 빈 커밋을 만들지 마세요.
- 모든 커밋 메시지는 Conventional Commits 형식을 사용하세요: `<type>(<optional-scope>): <description>`.
- 대표적인 `type`은 `feat`, `fix`, `docs`, `test`, `refactor`, `chore`, `build`, `ci`입니다.
- 제목은 변경 목적을 명확하고 간결하게 설명하고, 서로 무관한 변경을 하나의 커밋에 섞지 마세요.
- 호환성을 깨는 변경은 본문에 `BREAKING CHANGE:`를 기록하거나 `type!:` 형식을 사용하세요.

## 완료 기준

작업 완료를 보고하기 전에 다음을 확인하세요.

- 관련 테스트와 정적 검사가 통과했는가
- 실패 경로가 주문을 허용하지 않는가
- 재실행 시 중복 주문 가능성이 없는가
- 로그에 비밀정보가 없는가
- 문서가 실제 동작과 일치하는가
- 사용자가 요청하지 않은 실거래 범위가 추가되지 않았는가
- 초보 사용자가 내부 상태나 DB를 직접 수정하지 않고 작업을 완료할 수 있는가
- 실패 메시지가 원인, 보호 조치와 다음 행동을 한국어로 설명하는가
