# Input

- 2026-07-16 사용자 요청으로 `docs/TODO.md`의 미완료 항목과 대화에서 확인한 UX 개선을
  의존관계 순서로 모두 구현한다.
- 사용자는 목표 비중만 입력하고 하한·상한은 기본 혼합 밴드 정책으로 자동 계산한다.
- 설정은 개별 보유종목부터 시작하지 않고 자산군 목표와 자산군 내부 종목 구성으로 안내한다.
- 현재 보유하지 않은 종목도 검색·검증 후 목표 구성에 추가할 수 있게 한다.
- 토스 `getStocks`는 심볼 조회이므로 종목명 검색 카탈로그와 Toss 검증을 서버에서 분리한다.
- 토스 `buying-power`를 통화별로 수집하되 매수 가능 금액, 계좌 현금과 관리 현금을
  같은 값으로 간주하지 않는다.
- 사용자는 현재 고정 원화 금액 또는 현금 제외 정책을 명시적으로 선택한다. 브로커
  현금 전부 사용은 source of truth를 실계좌 표본으로 검증한 뒤 추가한다.
- Shadow 계획, Risk Gate, 주문 원장, Paper 체결, 장애 복구를 완성한 뒤 Live를 구현한다.
- Live는 기본 비활성 상태를 유지하고 단일 옵션이 아닌 다중 활성화 조건, 수동 승인,
  소액 한도, 킬 스위치, 멱등성, 주문 대사와 `UNKNOWN_BLOCKED`를 요구한다.
- 자동 검증과 개발 과정에서는 실제 계좌 주문을 전송하지 않는다. 실제 소액 주문은
  독립 설계 검토를 통과한 뒤 별도 실행 게이트로 다룬다.
- 기존 pnpm 모노레포에서 실제 토스증권 read-only 데이터를 사용한다.
- Next.js Web과 NestJS 11/Fastify adapter engine을 분리하고 두 앱을 Vercel Project로 운영한다.
- engine HTTP 계층을 Module, Controller, Guard와 singleton Provider로 현대화한다.
- Prisma 7과 PostgreSQL을 사용한다.
- 제품 런타임의 합성·더미 데이터는 제거한다.
- 주문 쓰기 API는 계속 하드 차단한다.
- 홈 외 5개 `준비 중` 자리표시자를 실제 라우트로 전환한다.
- 메뉴는 포트폴리오, 리밸런싱, 주문·기록, 문제 해결, 설정 순서로 구현한다.
- 계획·주문·복구 모델이 없는 상태를 가짜 데이터로 보완하지 않고 안전한 빈 상태로 표시한다.
- 기존 Prisma 목표 설정 모델을 사용해 목표 비중을 버전으로 저장하되 저장이 계획이나 주문을 만들지 않게 한다.
- Web 개발·production 포트를 `127.0.0.1:13000`으로 고정하고 host-run launchd로 유지한다.
- `home-server` Caddy와 Tailscale DNS를 통해 `https://stock.fredly.dev`로 노출한다.
